进程和线程

进程是程序的实例。大部分程序可以运行多个实例进程。

一个进程之内可以分为一到多个线程，一个线程就是一个指令流，将一条条指令交给cpu执行

在java中线程是最小的调度单位

并发和并行

并发：线程轮流使用cpu，同一时间应对多件事情的能力

并行：同一时间动手做多件事情的能力

sleep和 yield

sleep：

1、sleep会让线程从running进入阻塞状态

2、sleep被interrupt方法打断后，会抛出异常

3、睡眠结束后的线程未必会立刻得到执行

4、被打断的interrupt会 清空打断状态current.isInterrupted();

yield：

会让线程 从running进入runable就绪状态，然后调度其他线程

线程的优先级setPriority 并不能保证优先级高的先执行

join（）等待线程运行结束

join（time）最多等待时间

线程状态

从操作系统层面来说 五种

从javaApi层面来说 六种

根据 Thread.State 枚举，分为六种状态

  • NEW 线程刚被创建，但是还没有调用 start() 方法
  • RUNNABLE 当调用了 start() 方法之后，注意，Java API 层面的 RUNNABLE 状态涵盖了 操作系统 层面的 【可运行状态】、【运行状态】和【阻塞状态】（由于 BIO 导致的线程阻塞，在 Java 里无法区分，仍然认为 是可运行）
  • BLOCKED ， WAITING ， TIMED_WAITING 都是 Java API 层面对【阻塞状态】的细分，后面会在状态转换一节
  • TERMINATED 当线程代码运行结束

临界区

1、存在多个线程访问共享变量（本身没有问题，其实是读写操作指令发生交错，就会出现问题）

2、如果一个代码块中存在对共享资源的多线程读写操作，则这段代码为临界区

多个线程在临界区内执行，由于代码的执行序列不同而导致结果无法预测，称之为发生了竞态条件

Monitor 概念

java对象头 以32位虚拟机为例

其中 Mark Word 结构为

Normal是普通对象拥有hashcode，age（迭代年龄） biased_lock 偏向锁开关，0 1 状态

Biased是偏向锁，

Lightweight Locked 轻量级锁 会在栈帧中创建锁记录

Heavyweight Locked 重量级锁 每个重量级锁都会创建一个Monitor

Monitor 工作原理

每一个java对象都可以关联一个monitor对象。如果使用sy给对象上了（重量级）锁后，该对象的对象头，就会设置指向Monitor对象的指针。

如果不加sy，则根本不回去检查你的Mark Word 。

sy锁的优化原理

对使用者都是透明的，语法仍然是sy

重量级锁

自旋优化

当对象已经被上锁之后，线程并不会马上进入阻塞状态，而是会进行自旋重试，这样是为了避免进入阻塞队列后，进行上下文切换。

在多核cpu下自旋才能发挥优势，单核自旋就是浪费。java7之后不能控制自旋开关。默认开启。

轻量级锁

有多线程的访问，但是时间上是错开的 无竞争

通过在栈帧上创建锁记录（Lock Record）对象,内部可以存储对象的Mark Word。

Object reference指向锁对象，并尝试使用cas替换object中的Mark word，将其存入锁对象

如果cas替换成功，对象头中就会存储 锁记录的地址和状态 00，表示该对象已经被上锁

如果cas替换失败：

  • 如果是其他锁持有了该对象的轻量级锁，这表明有竞争，进入锁膨胀过程

  • 如果是自己执行了sy锁重入，则再添加一条LockRecord作为重入计数，取值为null

当退出sy锁的时候，如果是取值为null，则证明是重入，这时重置锁记录，表示重入计数减一。

当退出的时候，锁记录不为null，这时使用cas将mark Word的值恢复给对象头

成功，则解锁成功

失败，说明进行了锁膨胀（升级为重量级锁），此时进入重量级锁的解锁流程

锁膨胀

当给对象加轻量级锁的时候，发现已经被其他线程上锁了，此时进行锁膨胀。

为对象申请Monitor对象 owner指向lockRecord地址，让Object指向重量级锁地址，然后自己进入Monitor的EntryList Blocked

当轻量级锁的线程退出同步块解锁的时候，使用cas恢复失败。此时进入重量级锁解锁流程，按照Monitor地址找到Monitor对象，自身解锁完成。设置Owner为null，唤醒EntryList中的BLOCKED线程

偏向锁（独占锁）

轻量级锁在没有竞争的情况下，（只有单个线程使用）每次都重入仍然需要进行cas操作。此时引入了偏向锁进行优化。

用thread id 替换markword。之后如果发送了锁重入，就不用cas进行检查，直接看thread id即可

偏向状态

minitor对象头格式

  1. ，如果开启了偏向锁（默认开启）当一个对象创建的时候，markword后三位会变成101，这是thread、epoch、age都为0
  2. 偏向锁开启有延迟，可以加VM参数 -XX:BiasedLockingStartupDelay=0 禁用延迟
  3. 如果没有开启偏向锁，此时就是Normal，后三位001，前面都为0，hashcode第一次被用到才会赋值

撤销

  1. 调用hashcode 则会禁用这个对象的偏向锁，因为偏向锁的状态没有位置存储hashcode（轻量级锁在锁记录中存储hashcode，重量级锁在monitor中记录hashcode）
  2. 当有其他对象调用偏向锁的时候，会将锁升级成轻量级锁
    • ​ 批量重偏向：如果没有竞争，偏向了t1的线程有机会重新偏向t2，当只有一个线程访问20次后（撤销偏向锁阈值超过20次）jvm会认为是不是自己偏向错了，于是jvm会将对象重新偏向该线程
    • 当超过40次的时候，jvm会认为这个对象本来就不该偏向，于是整个类的所有对象，都会变成不可偏向的，新建的也不可偏向

wait/notifiy

原理

1、调用wait方法，即让线程进入monitor中的waitSet变为WAITING状态

2、WAITING线程会在owner线程调用notify或notifyAll时唤醒，但唤醒后并不意味着立刻获得锁，仍需要进入EntryList重新竞争

notify在waitSet中随机挑选一个唤醒，而notifyAll则唤醒在waitSet中所有等待的线程

wait、notify、notifyAll属于object对象的方法，必须获得此对象的锁，才能调用此方法

wait和sleep的区别

1、sleep是Thread方法，wait是Object的方法

2、sleep不需要强制和synchronized配合使用，但是wait需要和sy配合使用

3、sleep睡眠不会释放对象锁，wait在等待的时候，会释放对象锁

正确使用姿势

保护性暂停模式

即Guarded Suspension，用在一个线程等待另一个线程的执行结果

要点：

  • 有一个结果需要从一个线程传递到另一个线程，让他们关联同一个GuardedObject
  • 如果有结果不断从一个线程到另一个线程那么可以使用消息队列（见生产者/消费者）
  • JDK 中，join 的实现、Future 的实现，采用的就是此模式
  • 因为要等待另一方的结果，因此归类到同步模式

join原理

在t1线程调用t2.join，则t1需要等待t2运行结束后，才会继续执行

join的实现就是调用wait()方法 。其中的 wait(delay)的等待时间不是由传入的等待时间决定，而是记录了当前时间进行了处理，避免虚假唤醒。

exit（）方法会在run()方法执行后调用，他又会调用threadTerminated方法。

此方法会进行notifyAll（）唤醒所有等待的线程。

park&unpark

它们是 LockSupport 类中的方法

如果先执行了unpark，那么下次当线程park的时候，会直接运行，不会暂停。

原理：

先park后unpark

当调用了park方法后，会检查__counter的值,如果为0，则进入 _cond阻塞，然后设置 _counter=0;

当调用unpark方法后，先把counter=1；唤醒cond中的阻塞线程，线程恢复运行，把counter=0

先unpark再park

调用unpark，把counter=1；当线程调用park的时候，发现counter=1,则无需阻塞，继续运行，将counter=0；

死锁

使用jconsole工具查看，需要保证idea运行的jdk版本和jconsole版本一致

死锁产生的四个条件：

1、互斥：一个资源同一时间只能被一个线程访问

2、不可剥夺：当一个线程占用时，其他线程不可以抢夺该资源

3、请求与等待：当资源被一个线程占用时，其他线程只能等待资源的释放再去拥有

4、循环等待：多个线程形成环，都在等在其他线程释放

活锁

活锁出现在两个线程互相改变对方的结束条件，最后谁也无法结束

例如:游泳池一边放水，一边添水。谁都无法结束。

会造成饥饿：某个线程优先级太低，一直得不到cpu的调度执行

JMM内存模型

java定义的一套概念，底层对应着 CPU 寄存器、缓存、硬件内存、 CPU 指令优化等。

三大特性：

  • 原子性：保证指令不会受到线程上下文切换的影响
  • 可见性：保证指令不会受 cpu 缓存的影响
  • 有序性：保证指令不会受 cpu 指令并行优化的影响

可见性：

主线程尝试打断t线程。此时会发现，打断不了。因为 t 线程要频繁从主内存中读取 run 的值，JIT 编译器会将 run 的值缓存至自己工作内存中的高速缓存中， 减少对主存中 run 的访问，提高效率。

虽然主线程1秒后修改了这个值，但是t线程永远从缓存中读取，所以无法停下来。

如果在前面示例的死循环中加入 System.out.println() 会发现即使不加 volatile 修饰符，线程 t 也能正确看到

对 run 变量的修改了，因为 System.out.println()底层使用sy锁的。

注意 synchronized 语句块既可以保证代码块的原子性，也同时保证代码块内变量的可见性。但缺点是

synchronized 是属于重量级操作，性能相对更低

有序性：

JVM会在不影响正确性的前提下，可以调整语句的执行顺序。

这种特性称之为『指令重排』，多线程下『指令重排』会影响正确性。

cpu指令重排原理–为了提高效率

Volatile

能保证可见性、有序性，无法保证原子性。

底层原理是内存屏障，Memory Barrier

  • 对volatile变量的写指令后会加入写屏障，保证在该屏障之前，对共享变量的修改，同步到主存中
  • 对volatile变量的读指令后会加入度屏障，保证在该屏障之后，对共享变量的读取，加载主存中最新数据

写屏障会确保指令重排时，不会把屏障之前的代码排在写屏障之后

读屏障会确保指令重排时，不会把屏障之后的代码排在读屏障之前
